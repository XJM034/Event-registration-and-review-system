**创建日期**: 2026-03-14  
**当前状态**: 第六阶段已完成，待最终确认  
**文档目标**: 先统一移动端改造规则，再按优先级分批修复具体页面，避免“同类问题在不同页面反复出现”。

## 1. 本次校正结论

结合截图、现有实现和代码审查，原方案方向正确，但有 4 个需要修正的点：

1. **“20 个问题全部已确认”不准确**
   - 当前只能确认一批高优先级页面确实存在移动端问题。
   - 其余页面更适合归入“待排查清单”，不要直接当成已验证缺陷。

2. **项目管理页不是 Tabs 本身有问题**
   - `app/admin/project-management/page.tsx` 的 Tabs 已是 `grid-cols-1 sm:grid-cols-3`，基础响应式没问题。
   - 真正的问题在三个列表卡片内部：
     - `components/project-management/project-types-tab.tsx`
     - `components/project-management/projects-tab.tsx`
     - `components/project-management/divisions-tab.tsx`
   - 具体表现是：开关、编辑、删除操作区在移动端视觉层级混乱，按钮只有图标，触控面积不足。

3. **日志查询页不是简单“日期组件坏了”**
   - `components/security/security-audit-logs-viewer.tsx` 里的筛选器已经是单列堆叠。
   - 真正的问题是：
     - 日期范围输入框没有显式标签，在 iPhone Safari 上看起来像两个空白框
     - 快捷筛选、统计条、分页条、表格都还是桌面信息密度
     - 在 375px 到 768px 下可读性和可操作性都明显下降

4. **有些页面已经有初步移动端处理，不应重复改造**
   - `components/event-list.tsx` 已存在移动端卡片视图
   - `components/admin/admin-shell.tsx` 已存在移动端抽屉侧边栏
   - 这些页面更适合列入“第二轮优化”，先处理仍明显不可用的高优先级页面

## 3. 统一移动端改造规则

本次不采用“每个页面各写一套”的方式，而是统一遵循下面的规则。

### 3.1 断点策略

- 默认先写移动端样式，再用 `sm:` / `md:` / `lg:` 渐进增强
- **数据密度高的表格类页面**：优先在 `lg` 以下切换为卡片视图
- **普通表单和列表页面**：以 `sm` 为第一切换点

### 3.2 触控尺寸

- 移动端主要操作按钮高度不低于 `h-10`
- 输入框、选择器、日期控件高度优先使用 `h-11`
- 图标按钮若无文字标签，不能作为移动端主操作样式

### 3.3 筛选区规则

- 移动端默认折叠为“筛选条件”面板
- 日期范围必须带可见标签，不允许只出现两个无上下文空框
- 查询/刷新/分页大小在移动端分行展示

### 3.4 列表与卡片规则

- 桌面表格不直接缩小到手机宽度，而是切换为卡片
- 卡片结构统一为：
  - 顶部：标题 + 关键状态
  - 中部：摘要字段
  - 底部：全宽或双列操作按钮
- 卡片操作区与正文之间要有明确分隔

### 3.5 对话框规则

- 统一使用接近全宽的移动端尺寸：`w-[calc(100vw-2rem)]`
- 增加 `max-h` 和 `overflow-y-auto`
- 底部按钮在移动端垂直或双列排列，不保留过紧的横排

### 3.6 表单栅格规则

- 所有 `grid-cols-2` 在手机宽度下应降为 `grid-cols-1`
- 长文本说明、规则标签、统计信息允许换行，不强制单行显示
